## Lecture 4
- ## logical equivalence
  - Proving logical equivalence of two circuits
    - Derive the logical expression for the output of each circuit
    - show that these two expression are equivalent
  - **Method 1: truth table**
    - for every combination of inputs, if both expression yield the same output. Then that means they are equivalent.
    - Good for logical expressions with small number of variables.
  - **Method 2: algebraic manipulation**
    - with help of boolean algebra
- ## Logic expression simplification:
  - motivation: truth table method is not good enough. It could grow very large in size and become tedious as well as time consuming to write.
    - $n$ inputs means $2^n$ rows in the truth table
  - logic exxpression are better in this case
    - however there are equivalent logic expressions for the same truth table (?)
    - each of them lead to a circuit implementation
  - simplifying logic expressions leads to simpler and cheaper circuit design
    - because lesser components
  - There are many methods, algorithms and tools for simplifying logic expressions:
    - boolean algebra (math tool)
    - Karnaugh-maps (k-maps) (graphical tool)
- ## Boolean algebra revisit:
  - Boolean algebra is the algebra of logic 
    - deals with study of binary variables and logical operations
  - It transform logical statements into math symbols
    - calculate the truth or falsity of the statemenets
      - by using rules
  - [laws of boolean algebra](../../../notes/Boolean_algebra_laws.md) are powerful tools
    - to simplify the logic equations
- ## K-map:
  - K-Map is the graphical representation of the truth table or logic fucntion
  - In a K-Map, each cell represents one possible [minterm](../../../notes/minterm.md)
  - Cells are arranged following the **Gray Code** 
    - Gray code: two adjacent minterm in the table only differ by **one** variable
- ## K-map simplification rules:
  - find largest size groups of adjacent cells that are 1
  - it can only be:
    - $2^N (1, 2, 4, 8, ..)$ adjacent cells in each group
  - K-Map is toroid
    - like a circle
    - rightmost cells are considered adjacent to leftmost cells 
    - topmost cells are considered as adjacent to bottommost cells
  - larger groups 
    - fewer inputs to [`AND` gate](../../../notes/AND_gate.md)
  - Fewer groups
    - fewer `AND` gate and fewer inputs to [`OR` gate](../../../notes/OR_gate.md)
  - best group might not be unique

## ISA (instruction set architecture)
- ISA is the part of the processor that is both visible to the programmer and the compiler writer.
  - the native data types
  - instructions
  - registers
  - [addressing modes](../../../notes/addressing-mode.md)
  - memory architecture
  - interrupt and external handling
  - external I/O
- ISA includes the specification of the set of opcodes and the native commands implemented by a processor.
## The MIPS instruction Set
- MIPS (Microprocessor without Interlocked Pipeline Stages)
  - An example of RISC (Reduced Instruction Set Computer) design 
  - utilizes a small, high-optimized set of instructions
  - simple, clean MIPS processor whose architecture is easy to learn and understand.
  - RISC ISAs now have large share of embedded core in the market.
## ISA vs Assembly language
- ISA standardizes a public interface for the processor which should act as a basis to write programs.
- Assembly language is the term for a programming language.
- Ideally for each ISA, there is an Assembly language.
- But it is not uncommon that there is more than one or subtle variations of the Assembly language for a particular ISA.
- The Assembly language is defined by the Assembler.
## Why ISA
- If no ISA:
  - The "vocabulary" of the hardware can change from time to time.
  - No guarantee of how the instructions would look like in the next product.
- From programmers' point of view:
  - potentially need to reprogram everytime when we upgrade our computer.
- From system designers' point of view:
  - Hardware improvement may lead to incompatibility with existing applications.
- With ISA, all the above problems are solved:
  - All software developers have to do is conforming to the machine's ISA.
  - No need to worry about how hardware implements the instructions.
  - All system designers have to do is to ensure that the new processor implementations are backward compatible with ISA's definition, instead of new applications.
## Deisgn principles of (RISC) ISA:
- Simplicity favors regularity
  - {{Each instructions has a fixed number of operands in MIPS}}
  - Intel architecture supports a variation of number of operands
  - Regularity makes the implementation {{simpler}}
  - Simplicity allows a higher performance at a {{lower cost}}
- Smaller is faster
  - having a small enough number of registers leads to a faster processor
  - Larger number of registers, longer the electronic signal must travel
- Make common case fast
  - Constant operands occurs frequently
  - Encode constants inside arithmetic instructions,
    - much faster than if constants are loaded from the memory unit
- Good design demands good compromises

## Register operands
> MIPS data processing instructions operate on registers
>
  
- registers are very {{**fast temporary storage**}} on the processor used to hold variables
- The concept of variable vs register:
  - variables are logical storage, the amount is unlimited.
  - registers are actual physical storage, the amount is fixed on a particular processor.

## MIPS registers
- there are 32 general purpose registers on MIPS, each are 32 bits in size
  - Used for frequently accessed data (32-bit data is called a "word")
  - Numbered 0 to 31
  - Saved registers `$s0 - $s7` or `$16 - $23` (they are the same) are called the "Saved Registers" used to hold general variables of a program
  - Temporary registers `$t0 - $t7` or `$8 - $15` (they are the same) are called the "Temporary Registers: used to hold temporary variables of a program or during a particular processor of a program (since high level programming language like c++ and assembly are not exactly one-to-one)
  - `$zero` or `$0` register is called the "Zero register" used to hold just a constant value - 0

## How many registers are needed
- If:
  - too few: not enough to hold a large number of variables in a program
  - too many: leads to a complicated processor design, a longer clock cycle and lower the performance.
- need: good balance
  - provide enough number of registers 
  - keep the clock cycle short

## Data transfer in MIPS
- The main memory unit provides a very large storage
  - Large composite data like arrays, structures and dynamic data are all kept in the memory
- MIPS' design disallows data stored in memory to be manipulated directly
  - Data must be first transferred to registers are manipulation 
  - and then transfer back from registers to the memory.
- the following two instructions are exactly designed for this purpose:
  - `lw` (load a word) moves data from memory to a register
  - `sw` (store a word) moves data from a register to a specific memory location

- Memory is a consecutive arrangement of storage locations

  - Most architecture address individual bytes

## Immediate operands
- To do an expression like `reigster1 = register2 +/- constant`
- `addi` means "add immediate" (immediate is the constant)
  - constant part is always the last operand
- Zero register:
  - `$zero` or `$0` register holds a constant 0, read-only, can't be overwritten
  - we can play some tricks using it and `add`, `addi` instruction:
    - move between registers:
      - `add $t0, $0, $s0`
    - initialize a register to a constant:
      - `addi $t0, $0, 5`
  
## Memory vs Register vs Constant
- Major difference is the instruction's execution time.
  - **Memory** is outside the processor, far from the processing unit
    - Memory operand takes a long time to load and store
  - **Register** is inside the processor, close to the processing unit
    - Register operand takes a short time to get the value 
  - **Constant** already encoded in the instruction 
    - Constant operand value is immediately available (that's probably why they are called the **immediate**)
  - A program is a mixture of these 3 types of operations

## Logical Operations
- `and $rd, $rs, $rt` 
- `or $rd, $rs, $rt` 
- `nor $rd, $rs, $rt` (not-or)
- these are all bitwise logical operations act on each bit of `$rs` with the corresponding bit of `$rt`
- the results are stored in `$rd` 
- `andi` and `ori` are the immediate version

## shift
- `sll $rd, $rs, shamt`
- `srl $rd, $rs, shamt`
- they are "shift left logical" and "shift right logical" 
- the `shamt` is the shift amount
- what gets shifted is `$rs`, result stored in `$rd`
- `sll` is basically multiplication by $2^k$ where $k$ is `shamt`
- `srl` is basically division by $2^k$ where $k$ is `shamt`

## "Less than" test
- Beside testing the equality, we are often also interested in knowing whether a variable is **less than** or **greater than** other variables.
  - e.g. exit a loop when an array index is less than a specific variable
- `slt` ("set less than")
  - `slt reg1, reg2, reg3`
  - set `reg1` to be 1 if `reg2` is less than `reg3`
  - otherwise, set it to 0
- `slti` ("set less than immediate")
  - `slt reg1, reg2, constant`
  - set `reg1` to be 1 if `reg2` is less than the constant value.

## branch instruction desgin
- MIPS compilers use `beq, bne, slt, slti` and the fixed value 0, which is always available in `$zero`
- to create all comparison operations
- in other words, any comparsion operations can be created with the above 4 MIPS instructions
  - equal, not equal
  - less than, less than or equal
  - greater than, greater than or equal
- why not `blt` (branch less than), `bge` (branch greater or equal), etc.
  - hardware for `<`, `>=` is slower than `=` and `!=`
  - combining with branch involves more work per instructions. Requires a slower clock or longer clock cycle.
  - All instructions penalized. Since The D-flip-flops in the processor for updating things needs to account for the longest clock cycle of an instruction.
  - beq and bne are the common cases

## Branch with zero
- Branch on greater than or equal to zero
  - `bgez $s, label` ; `if ($s >= 0)`
- Branch on greater than zero
  - `bgtz $s, label` ; `if ($s > 0)`
- Branch on less than or equal to zero:
  - `btez $s, label` ; `if ($s <= 0)`
- Branch on less than zero:
  - `bltz $s, label` ; `if ($s < 0)`

## "Jump register" instruction
- another unconditional jump instruction
  - `jr` ("jump register")
    - e.g. `jr reg`
    - jump to the 32-bit address specified by the register: `reg`
- it is usually used for **procedure call** or **case/switch statement**

## Implementating MIPS instructions
- The computer "see" the instruction as machine language or machine code encoded in the binary.
- MIPS instructions
  - Encoded as 32-bit instruction words
  - Broken up into a number of fields
  - Small number of formats encoding operation code (opcode), register numbers.
  - Regularity
- For the details, see [instruction encodings](../../../notes/instruction-encoding.md)

## Design principle #4 (again)
- Good design demands good compromises
- Ways to encode instructions:
  - **variable length** or **fixed length**
- How to choose
  - Use **variable length** to optimize code size to save storage space
  - Use fixed length to optimize performance and reduce complexity
- The compromise that MIPS had chosen is to have fixed length instructions
  - so that the fetch and decode stage are faster and with simpler hardware design.

## Other important MIPS instructions
### Unsigned arithmetic
- `addu $rd, $rs, $rt`
  - "add unsigned"
  - addition ignoring overflow
- `subu $rd, $rs, $rt`
  - "subtract unsigned"

### Load/store a byte
- `lb $rt, offset($rs)`
  - **sign extend** to 32 bits in `$rt`
- `lbu $rt, offset($rs)`
  - **zero extend** to 32 bits in `$rt`
- `sb $rt, offset($rs)`
  - store the rightmost byte in `$rt` to `$rs`

### Logical operations
- `nor $rd, $rs, $rt`
  - "not or" bitwise operation
  - Useful to invert bits in a word. `a NOR b == NOT (a OR b)`
  - `nor $t0, $t1, $zero` simply does a `NOT` operation to invert the bits

### Shift left/right logical variables
- `sllv $rd, $rt, $rs` ; `$rd = $rt << $rs`
- `srlv $rd, $rt, $rs` ; `$rd = $rt >> $rs`

## Supporting Procedures in Computer hardware
- Procedures (also called subroutines) are necessary in any programming language.
- they allow better structuring of programs
- Thus we need the mechanism to jump to other procedures and return from it.

## Supporting prodecures in computer hardware
- necessary steps for executing a procedure:
  - Place the parameters in place where the procedure can access them.
  - Transfer control to the procedure
  - Acquire the storage resources needed for the procedure

## Principle of locality
> Programs access a small proportion of their address space at any time
>
> 

- Such execution pattern exhibits two types of localities 
- **temporal locality**
  - items accessed recently are likely to be accessed again soon
  - e.g. instructions in a loop, induction variables
- **Spatial locality**
  - items near those accessed recently are likely to be accessed soon
  - e.g. sequential instruction access, array data
- This property is the KEY to memory hierarchy
- Temporal and spatial locality can be observed in both instructions & data

## Memory hierarchy
- A **memory hierarhy** consists of multiple levels of memory
- Users have the illusion of a memory 
  - as large as the largest level 
  - as fast as the fastest level 
- A trade-off between performance and cost

![alt text](attachments/image-9.png)

- how does memory hierarchy operate:
  - Store everything on hardisk 
  - when program starts to execute
  - inititally, 
    - instructions and data are loaded into **memory(DRAM)** from disk 
  - Upon first caccess, 
    - A copy of the referenced instruction or data item (a block of them) is kept in **cache**
  - In subsequent accesses,
    - first, look for the requested item in the cache
    - if the item is in the cache, return the item to the CPU 
    - if not in the cache, look it up in the memory
  - if not found in this levle, look it up at next level until found 
  - Keep (or **cache**) a copy of the found item at this level after use.

- hit or miss
  - Data are copied in blocks (aka line, unit of copying, maybe multiple words) between only two adjacent levels at a time
  - If accessed data is present is upper level
    - **Hit**: access satisfied by upper level 
      - Time to send data outL **hit time**
  - if accessed data is absent
    - **Miss**: block copied from lower level 
      - data transfer occurs from the lower level to the upper level
    - Time needed: **miss penalty**
- Performance Measures
  - **Hit rate** (or **hit ratio**):
    - Fraction of memory accesses found in the upper level
  - **Miss rate**: 1 - hit rate
  - Cache hit time: 
    - is the time to determine miss or hit + time to access the cache
  - Cahce miss penalty:
    - is the time to bring a block from lower level to upper level
  - Hit time << Miss penalty

- Cache performance

> Average memory access latency = hit time + miss rate * miss penalty
>

- For a configuration with a single level of cache, average memory latency
  - = hit time + miss ratio * miss penalty
  - = hit time + (1 - hit ratio) * miss penalty
- For two levels of caches, average memory latency:
  - = hit time1 + miss ratio1 * miss penalty1
  - = hit time1 + miss ratio1 * (hit time2 + miss ratio2 * miss penalty2)
- The same idea can be extended to multiple levels of caches

## cache 
- Cache operations

    - where is a block placed in the cache
  - Block identification

    - when a write occurs, is the information written only to the cache
- Cache Organization
  - Cache is organized as an array of cache blocks 
  - Each memory location is mapped to ONE location in the cache
  - Cache block
    - A minimum unit of information that can be present in cache or not
  - Cache block sizes
    - commonly used today: 32 bytes and 64 bytes
  - Three basic organizations 
    - Direct-mapped 
      - one memory block to one possible cache block 
    - Set-associative
      - one memory block to one set of possible cache blocks 
    - Fully-associative
      - one memory block to all possible cache blocks
- Direct mapped cache
  - When transfer a block from main memory to cache 
    - its location in cache is determined by memory address
  - Direct mapped: only one candidate location in cache
    - (Block address) % (# of blocks in cache)
  - it's a Many-to-one mapping
  - ![alt text](attachments/image-10.png)

- Block Placement example
  - Block size = 8 words (32 bytes) 
  - Direct mapped cache has 8 blocks
- Example: `lw $t0, 1200($zero)`
  - Processor asks for a word from byte address `1200`
  - that word resides in block 37 in memory, e.g. floor(1200/32) = 37
  - The whole block will be loaded to cache set 5, i.e. 37 mod 8 = 5
  - Then that word will be sent to processor
- Shortcut:
  - byte address 1200: 
    - ` 0000 0000 0000 0000 0000 0100 | 101 | 1 0000 `
  - It belongs to memory block 37: 
    - which is the address above after shifting after the last 5 bits. Essentially taking the floor of the division by 32
    - The shifted away bits are important. It indicates the position of the starting address of that specific targeted word inside the 37th block (again a block is 32 bytes not bits)
    - i.e. 37 * 32 + `<shifted away 5 bits>` = the desired word address
    - after shitfing away the 5 bits
      - the current last 3 bits is the exact index in the cache
      - the remaining bits is called the tag, the block identifier 
  - Block 36 will be loaded to cache block 5: `101`
  - Shortcut:
    - if the number of cache sets (N) is a power of 2; N = 2^n
    - cache location = the low order m bits of the block address (the address after shifting away the least significant 5 bits is called the block address)
- Block Identification: Tags and Valid Bits
  - How do we know which particular block is stored in a cache location
    - Store block address as well as the data 
    - actually, only need the high-order bits 
    - which is the **tag** 
  - What if there is no data in a location
    - valid bit: 1 = present, 0 = not present
    - initially 0

![alt text](attachments/image-11.png)

- Bits in Cache
  - Question: A direct-mapped cache with 16KB of data and 8-word (2^5 bytes) blocks
  - Assume 32-bit memory address
  - What is the actuall cache size
- Solution:
  - 16KB = 2^14 bytes 
  - each block contains 2^5 bytes of data, so the block size is 2^5
  - so there have to 2^14 / 2^5 = 2^9 many blocks (also this many sets since it is direct-mapped) in the cache
  - so the cache sets can be indexed by 9 bits, the index field in the 32-bit address is 9 bit
  - again the block size is 2^5 = 32 bytes. So the bits shifted away is also 5 (the targeted address divided by 32)
  - so what have to be remembered in the cache is the:
    - 1 bit called the valid bit
      - indicating whether the data is present
    - 32 - 9 - 5 (9 is cache block index, 5 is shifted away btis, aka the position of the data) = 18 bits of tag field to identify the block 
    - data: 8 (bits per byte) * 32 (# of bytes per block) = 256 bits of data
  - so then just multiply all the bits that have to be stored by the number of blocks (2^9), done
  - so the answer is The cache size is exactly: `2^9 * (1 + 18 + 256)` bits

- Disadvatnage of direct mapped cache
